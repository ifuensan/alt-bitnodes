## Requirements
### Requirement: The installer provisions a pool of Tor instances

`install.sh` SHALL provision additional Tor instances (`tor@bitnodes1` through
`tor@bitnodes5`) alongside the distribution's default Tor service, each with
its own DataDirectory and a distinct localhost SocksPort (9051–9055).
Provisioning SHALL be idempotent: re-running the installer on a host where the
instances already exist SHALL NOT recreate, break, or duplicate them.

#### Scenario: Fresh install provisions the pool
- **WHEN** `install.sh` runs on a host without the extra Tor instances
- **THEN** `tor@bitnodes1`..`tor@bitnodes5` exist, are enabled, and are
  running, each listening on its own SocksPort (127.0.0.1:9051–9055)

#### Scenario: Re-run is a no-op for existing instances
- **WHEN** `install.sh` runs on a host where the instances already exist
- **THEN** the instances keep their DataDirectory and configuration, and the
  installer completes without error

### Requirement: The crawler spreads onion dials across all Tor instances

The crawler configuration generated by `install.sh` SHALL list every Tor
SocksPort (the default 9050 plus all pool ports) in `tor_proxies`, so that
`.onion` connection attempts are distributed across the pool.

#### Scenario: tor_proxies lists the whole pool
- **WHEN** `install.sh` generates `crawl.f9beb4d9.conf`
- **THEN** `tor_proxies` contains `127.0.0.1:9050` through `127.0.0.1:9055`

### Requirement: Onion peers are fully sampled

The crawler configuration SHALL set `onion_peers_sampling_rate = 100` so that
every gossiped `.onion` address is eligible for crawling.

#### Scenario: Sampling rate restored
- **WHEN** `install.sh` generates `crawl.f9beb4d9.conf`
- **THEN** `onion_peers_sampling_rate` is 100

### Requirement: Rollout has explicit health checks and a one-commit rollback

The change SHALL document post-deploy health checks — host load average,
Tor circuit-drop messages in the journal, and onion counts in fresh
snapshots — and a rollback that consists of reverting a single commit
(restoring `tor_proxies = 127.0.0.1:9050` and `onion_peers_sampling_rate =
25`). Idle pool instances left behind by a rollback SHALL be harmless.

#### Scenario: Saturation is detected and reverted
- **WHEN** post-deploy checks show sustained load near vCPU count or Tor
  journals flooding with dropped-circuit messages
- **THEN** reverting the change commit and letting CI redeploy restores the
  previous single-proxy, 25%-sampling behaviour without manual host surgery
